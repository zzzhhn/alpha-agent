# Alpha Core UX Audit (2026-05-23)

## Methodology

This audit applies 10 UX principles (5 base + 5 quality differentiators) across 5 dashboard pages. Each (page, principle) cell is graded with a severity (Critical / High / Medium / Low / Pass), a concrete violation citing file:line where possible, a one-sentence fix, and an implementation complexity estimate. Read-only audit; no source files were modified. Violations are evidence-based: every entry traces to a specific component behavior, not a stylistic preference.

## Recommended starting point

**/alpha should be redesigned first.**

The weighted violation score (17) leads all pages by a significant margin (next is /backtest at 15), but more importantly the violations compound each other. A user encountering `/alpha` for the first time faces: a blank page with no onboarding signal (Principle 7), a textarea with no affordance cue that submission triggers a 5-30 second LLM call (Principle 3 Critical), then after submit, a wall of 9 sequentially-rendered panes with no visual hierarchy to guide "what do I read first" (Principle 2 High). The single Critical violation (no in-pane loading feedback during translation) means that the central interaction of the page, which is the LLM translation, provides no feedback during the wait. For a page built around a 5-30 second async operation, that is the highest-impact gap in the product.

The redesign's top 5 required elements are: (1) In-pane loading state during `translateHypothesis` (a spinner or animated placeholder inside HypothesisInputPane, not just a subbar pill); (2) Post-result pane collapsing so only TRANSLATE.SPEC + SMOKE.PROBE + BACKTEST.QUICK are visible by default, with the 5 analytics panes behind a "Show analysis" toggle; (3) A single dominant CTA in the result state (either "Run Backtest" if the smoke IC is healthy, or "Re-translate" if degenerate); (4) Relocate the tips text to below the textarea rather than right-aligned next to an unrelated dropdown; (5) Deprecate the LLM.PROVENANCE pane when no BYOK key is configured (it reads as a broken pane showing "platform default / ---" to the majority of users).

## Cross-page patterns

**No global Toast system.** Every page handles async operation feedback differently. `/picks` uses a `ToastState` machine local to `RefreshButton.tsx`. `/alpha` uses inline `<p className="text-tm-neg">` inside panes. `/backtest` uses a dedicated `TmPane title="ERROR"`. `/factor-lab` uses per-row `rowErrors` state in the proposals table. `/evolution` has no mutation feedback at all (actions trigger `router.refresh()` silently). This means 4 out of 5 pages have incoherent feedback UX for failed async operations, and 0 out of 5 pages notify the user of a successful action (the Approve on `/factor-lab` simply refreshes the table with no "approved" confirmation). The fix is architectural: a single `useToast` hook + `<ToastProvider>` in the dashboard layout, with a standardized `toast.success(...)` / `toast.error(...)` call replacing all inline error state. This is a one-time layout-level addition.

**No loading.tsx on 3 of 5 pages.** Only `/picks` has a `loading.tsx` skeleton. `/alpha`, `/backtest`, `/factor-lab`, and `/evolution` all show blank or partially-hydrated states during their initial RSC fetch or client hydration. The common fix is route-level `loading.tsx` files for each remaining page, mirroring the `/picks` pattern which correctly provides an `animate-pulse` skeleton matching the final layout structure. This should be implemented as a single ticket covering all 4 missing files before any page-level redesign begins, since it requires no design decisions and directly addresses Principle 3 (Visibility of system status) for all pages simultaneously.
